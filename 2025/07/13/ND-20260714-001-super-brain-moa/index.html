超级大脑：让多个大模型协作思考

本文最后更新于 2026年7月14日 晚上

之前用单个大模型写技术文章，总遇到一个魔咒：开头写得像人话，越到后面越像机器翻译。更反直觉的是，换了个更大的模型，问题依旧——代码块缩进错乱、引用格式偶尔缺方括号、逻辑跳跃时自己都没意识到。

折腾了一周才想明白：问题不在于模型不够大，而在于单一模型的能力分布本身就是不均匀的。就像让一个程序员同时写代码、审代码、写文档，总有一项会拉胯。后来换了思路，让多个模型分工协作，输出质量明显提升。今天聊聊怎么从论文原理到工程落地，把”多模型协作”这件事做顺。

前置条件

  • 硬件：双卡 RTX 2080 Ti 魔改版（单卡 22 GB，双卡共 44 GB），Turing 架构
  • 服务器：ModelBase（ModelBase 服务器）
  • 推理引擎：llama.cpp，models-preset 多模型模式
  • 代理层：new-api（ModelBase 服务器0:3000）
  • 模型：Qwen3.5-35B-A3B-Ornith（Q4_K_M，实测 104 t/s）、Qwen3.6-35B-A3B（Q4_K_M，实测 103 t/s）

单模型的天花板在哪

单个大模型处理复杂任务时，有三个天然局限：

注意力稀释。上下文越长，模型对关键信息的关注度越低。写长文时，前面的论点容易在后面被”遗忘”。

角色混乱。同一个模型既要当创作者，又要当审稿人，结果往往是”自己夸自己”。检查环节流于形式。

能力边界。不同模型在不同任务上有专长差异。一个模型擅长的事，另一个可能恰恰相反。让一个通用模型做所有事，不如让专业模型各司其职。

更关键的是，论文已经证明：把多个中等模型结合起来，效果能超过一个更大的模型。这不是玄学，而是结构化的互补性利用。

MoA 论文的核心思想

Mixture-of-Agents（MoA）论文提出了一个简洁的框架：分层聚合。

核心流程就三步：

  1. 并行生成。多个基础模型同时处理同一个输入，各自输出中间结果。
  2. 分层聚合。把多个输出拼接后，交给聚合模型做评估、排序和增强。
  3. 最终生成。用聚合后的结果作为上下文，让聚合模型生成最终答案。

关键洞察在于：不同模型对同一问题的”盲区”往往互补。模型 A 没注意到的细节，模型 B 可能抓得很准。分层聚合就是把这种互补性结构化地利用起来。底层模型负责发散，顶层模型负责收敛——既不丢失多样性，又能保证输出质量。

moa-plus 的工程实现

论文框架是骨架，工程实现才是血肉。moa-plus 在这个基础上做了几个关键优化：

标准化接口。所有模型统一走 OpenAI 兼容接口，底层可以是 llama.cpp、vLLM、甚至云端 API。换模型不需要改业务逻辑。

预设方案。针对不同任务类型预定义了 4 种模型组合（default/research/fast/creative），用户通过 --preset 参数选择，无需了解底层模型细节。

容错机制。参考模型超时或报错时自动跳过，只要至少 1 个参考模型有输出就继续。聚合器失败时 fallback 到最长的参考响应。

具体到我本机的架构，分三层：

底层：llama.cpp 的 models-preset 模式管理多个模型实例，每个模型独立加载到显存。双卡 44 GB 总显存刚好能同时跑两个 35B Q4_K_M 量化模型。

中间层：new-api 代理层（ModelBase 服务器0:3000）负责路由。用户请求进来后，代理层根据预设方案决定调用哪些模型、怎么聚合。

顶层：generate 黑盒接口。用户看到的和调普通模型一样，只是底层在偷偷做多轮协作。

generate 黑盒模式设计

核心设计原则：对上层透明，对下层可控。

用户调用 generate(prompt)，返回一个结果。中间发生了什么，用户不需要知道。但开发者可以通过配置调整协作策略。实现上，generate 函数做了这些事：

  1. 接收 prompt 和预设方案
  2. 参考模型并行调用（ThreadPoolExecutor，各自独立超时）
  3. 收集所有参考输出
  4. 聚合器模型自动融合所有回答 → 生成最终文本
  5. stdout 输出纯文本（或 fallback 到最长参考响应）

关键是把聚合策略和模型调用解耦。换模型不需要改聚合逻辑，换策略不需要换模型。

实际调用长这样：

返回的是一段纯文本，和普通模型的输出完全一致。内部是参考模型并行生成 + 聚合器自动融合，对调用方完全透明。业务代码不需要维护多套模型配置，一行命令搞定。

四种预设方案

针对不同场景，moa-plus 内置了四种预设方案：

default（通用默认）。GLM-5 聚合器 + DeepSeek-V4-Flash + Sensenova 参考模型。参考模型并行生成，聚合器融合后输出。日常任务推荐，兼顾质量和速度。。适合简单问答、分类任务。延迟最低，成本最低。

research（深度分析）。glm-5.2 聚合器 + DeepSeek-V4-Pro + GLM-5.1 + Sensenova 参考模型（共 3 个参考模型）。适合需要高质量输出的场景，比如技术调研、方案评估。。

fast（快速响应）。glm-5-turbo 聚合器 + DeepSeek-V4-Flash 参考模型（仅 1 个参考）。延迟最低，适合简单任务快速过一遍。。

creative（创意写稿）。DeepSeek-V4-Pro 聚合器 + GLM-5.1 + Sensenova + Qwen3.5 参考模型（共 3 个参考）。写文章、创意输出专用，融合多种文风。。

实战：博客写稿 + 审稿集成

把 moa-plus 集成到博客工作流后，写稿和审稿变成两个独立的 API 调用环节。本地目录 ~/projects/blog 下的脚本自动触发协作流程，写完直接预览。

写稿阶段用 creative 预设（创意融合）：Qwen3.5-35B-A3B-Ornith 负责初稿生成，另一个模型负责结构优化。实测写 2000 字草稿，总耗时约 3-5 分钟（含参考模型并行生成 + 聚合器融合），比单模型多花 5 秒，但质量明显提升。多个模型同时构思不同角度的内容，聚合后选出最佳框架。

审稿阶段用 research 预设（深度分析）：一个模型先审，指出问题；另一个模型针对审出来的问题给出修改建议。两个角色分开，避免”自己审自己”。这样审稿不会放过明显错误，也不会过度吹毛求疵。

集成到 Hexo + Fluid 博客工作流后，实际效果是：初稿质量明显提升，审稿环节发现的问题从”格式问题”升级成了”事实错误”和”逻辑漏洞”。

踩坑记录

坑一：显存识别错误

现象：llama.cpp 加载模型时报 OOM，但算了一下显存明明够用。

根因：系统把魔改版 2080 Ti 识别成了官方标称规格，实际单卡是 22 GB（魔改后）。

解法：在 llama.cpp 启动参数里手动指定 --n-gpu-layers 和显存分配，不依赖自动检测。

坑二：聚合模型太弱导致噪音

现象：多模型协作后输出质量反而下降，比单模型还差。

根因：聚合模型能力不够，多模型的多样性在它手里变成了噪音。

解法：聚合层必须比基础层强，至少不能弱。换用更强的模型做聚合后问题消失。

坑三：同质模型堆数量没用

现象：用了 4 个基础模型，效果还不如 2 个。

根因：4 个同质模型不如 2 个异质模型。模型专长不互补时，堆数量没有意义。

解法：选专长互补的模型组合，效果远好于单纯加数量。

效果与反思

跑了两周，几个观察：

质量提升明显。审稿阶段能发现的问题类型比单模型更丰富（主观体感，未做精确统计）。写稿时逻辑跳跃的情况减少了，因为角色分工让不同模型各管一块。

速度是硬约束。多模型协作的总耗时 = 参考模型并行阶段（取最慢的）+ 聚合器融合阶段。实测 creative 预设写 2000 字约 3-5 分钟，比单模型慢但质量提升明显。

方案选择比模型选择更重要。同样是两个 35B 模型，用并行投票和用串行辩论，效果差异比换模型还大。这说明协作策略是杠杆，模型只是原材料。

缓存是性能关键。对于重复性任务，缓存命中率直接影响实际体验。

黑盒模式确实好用。上层调用方完全不需要知道底层有多少模型在协作，就像调一个普通 API。每次换模型不需要改业务代码。

最大的反思是：多模型协作不是银弹。简单任务用单模型更快更便宜。只有当任务需要多种能力协同，或者对质量要求极高时，多模型才有优势。
