Voice of Customer (VoC) 技能

技能名称：Voice of Customer (VoC)
简称：VoC
版本：1.1
创建日期：2026-03-23
最后更新：2026-03-23
维护人：xiaorui

📚 技能概述

VoC（Voice of Customer）技能用于记录、分析和响应用户（客户）反馈，将用户的声音转化为可执行的决策和文档更新。

核心功能

  1. 如实记录用户发言

    • 记录时间戳
    • 记录发言内容
    • 记录会话上下文

  2. 发掘有价值信息

    • 分析用户需求
    • 识别潜在问题
    • 发现改进机会
    • 更新相关文档

🎯 适用场景

触发条件

当以下情况发生时，激活VoC技能：

  1. 用户反馈意见：

    • “我觉得…”
    • “我希望…”
    • “建议…”

  2. 用户提出需求：

    • “需要…”
    • “想要…”
    • “能否…”

  3. 用户遇到问题：

    • “问题…”
    • “错误…”
    • “不行…”
    • “失败…”

  4. 用户表达偏好：

    • “喜欢…”
    • “不喜欢…”
    • “习惯…”

非适用场景

以下情况不需要VoC技能：

  1. 简单询问：

    • “今天天气怎么样？”
    • “帮我看下…”

  2. 命令执行：

    • “重启服务”
    • “发布博客”

  3. 临时对话：

    • 无实质内容的寒暄
    • 简单的确认回复

🔄 工作流程

1. 记录阶段（实时）

记录位置：memory/voc/voc-log.jsonl

记录格式：JSONL（每行一个JSON对象）

2. 分析阶段（实时/定期）

分析维度

维度 说明 示例
需求识别 识别用户的明确需求 “需要添加xxx功能”
问题发现 发现用户遇到的问题 “xxx功能不好用”
偏好提取 提取用户的偏好信息 “我喜欢简洁的界面”
改进建议 记录用户的改进建议 “建议xxx改成xxx”
情感分析 分析用户的情感倾向 积极/消极/中性

价值评分

基于以下维度评分（1-10分）：

  1. 重要性（1-3分）：

    • 战略级需求：影响长期方向、核心竞争力 → 3分
    • 核心功能需求：影响主要功能 → 3分
    • 次要功能需求：影响部分用户 → 2分
    • 一般性建议：优化性建议 → 1分

  2. 可行性（1-3分）：

    • 容易实现：现有能力可直接实现 → 3分
    • 需要设计：需要设计方案 → 2分
    • 需要重构：需要大幅调整 → 1分
    • 暂不可行：当前条件下无法实现 → 0分（但仍记录）

  3. 影响范围（1-4分）：

    • 全局影响：影响多个模块、整个系统 → 4分
    • 模块影响：影响单个主要模块 → 2分
    • 局部影响：影响特定功能 → 1分

总分 = 重要性 + 可行性 + 影响范围

评分示例：

场景 重要性 可行性 影响范围 总分 评级
战略级需求 + 易实现 + 全局影响 3 3 4 10 🏆 极高价值
核心功能 + 需要设计 + 模块影响 3 2 2 7 ⭐ 高价值
次要功能 + 容易实现 + 局部影响 2 3 1 6 🌟 中等价值
一般建议 + 需要重构 + 局部影响 1 1 1 3 💡 低价值

评级标准：

  • 9-10分：🏆 极高价值 → 优先处理
  • 7-8分：⭐ 高价值 → 近期规划
  • 5-6分：🌟 中等价值 → 中期规划
  • 3-4分：💡 低价值 → 暂存档
  • 1-2分：📝 信息性 → 仅记录

标签使用规则：

  1. 主标签优先：每个记录至少1个主标签（反馈类型）
  2. 次标签补充：根据情况添加情感标签、影响标签
  3. 标签数量控制：每条记录最多3-5个标签
  4. 标签一致性：使用标准化标签，避免同义词混用

3. 更新阶段（按需）

更新目标

根据分析结果，更新以下文档：

  1. MEMORY.md：

    • 添加新的用户偏好
    • 记录重要的需求
    • 更新相关规则

  2. 规则文件：

    • 更新 USER.md：添加新的偏好
    • 更新 SOUL.md：调整行为方式
    • 更新 AGENTS.md：增加新的工作规范

  3. 任务记录：

    • 创建任务记录：memory/tasks/YYYY-MM-DD-Task-XXX.md
    • 跟踪需求实现进度

  4. VoC汇总：

    • 定期生成VoC汇总报告
    • 位置：memory/voc/voc-summary.md

更新策略（减少打扰）

重要原则：

  • 后台静默处理：分析在后台进行，不主动推送每次分析结果
  • 按需推送：只在必要时主动告知用户（如发现重要问题、需要用户决策）
  • 批量汇总：定期生成汇总报告，一次性告知用户
  • 例外情况：用户主动询问时，才提供实时分析结果

推送判断标准：

情况 是否推送 说明
用户主动询问 ✅ 是 用户明确要求查看分析结果
发现关键问题 ✅ 是 影响核心功能或安全的问题
发现高价值需求 ✅ 是 价值评分 ≥ 9分的重大需求
定期汇总 ✅ 是 每日或每周汇总时推送
普通分析结果 ❌ 否 常规分析记录到日志即可

🏷️ 分类体系

分类维度（理论+实践）

1. NPS情感分类（基于Net Promoter Score理论）

分类 英文 评分范围 特征 应对策略
推广者 Promoters 9-10 极其满意，推荐给他人 🔔 请求推荐、分享成功案例
被动者 Passives 7-8 满意但缺乏热情 🔄 了解原因、提升体验、转化为推广者
贬损者 Detractors 0-6 不满意，可能批评 ⚠️ 优先解决问题、深入理解痛点、重建信任

适用场景：用户表达整体满意度或推荐意愿时

2. 反馈类型分类（基于实际产品反馈）

分类 子分类 英文 示例 处理优先级
功能需求 新功能 Feature Request “希望添加xxx功能” 🔴 高
功能改进 Improvement “xxx功能可以优化” 🟡 中
功能删除 Removal “xxx功能不需要” 🟢 低
问题报告 Bug Bug Report “xxx报错了” 🔴 高
性能问题 Performance “xxx很慢” 🔴 高
兼容性问题 Compatibility “xxx不兼容” 🟡 中
体验反馈 易用性 Usability “xxx不好用” 🟡 中
界面设计 UI/UX “xxx界面很乱” 🟡 中
交互流程 Workflow “xxx流程太繁琐” 🟡 中
偏好表达 喜好 Preference “我喜欢xxx” 🟢 低
不喜欢 Dislike “我不喜欢xxx” 🟢 低
习惯 Habit “我习惯xxx” 🟢 低
改进建议 优化建议 Optimization Suggestion “建议xxx改进” 🟡 中
创新想法 Innovation Idea “可以考虑xxx” 🟡 中
其他 无法归类 Other “xxx” 🟢 低

优先级说明：

  • 🔴 高：影响核心功能或大量用户的问题
  • 🟡 中：影响体验或部分用户的问题
  • 🟢 低：偏好、习惯、非功能性建议

3. 业务影响分类（基于对业务的影响）

分类 英文 影响维度 示例 价值评分
战略影响 Strategic Impact 影响长期方向、核心竞争力 9-10分
产品影响 Product Impact 影响产品功能、性能、质量 7-8分
用户体验影响 UX Impact 影响用户满意度、使用习惯 5-6分
运营影响 Operational Impact 影响工作效率、流程 3-4分
无影响 No Impact 一般性建议、闲聊 1-2分

4. 数据源分类（基于反馈来源）

数据源 英文 特征 可靠性 分析方法
直接反馈 Direct Feedback 用户主动表达，意图明确 ⭐⭐⭐⭐⭐⭐ 直接分析
间接反馈 Indirect Feedback 用户通过行为或第三方表达 ⭐⭐⭐⭐ 需要推断
隐性需求 Latent Needs 用户未明确表达但实际需要 ⭐⭐ 数据分析

5. 生命周期分类（基于产品发展阶段）

分类 英文 说明 处理方式
发现阶段 Discovery 新用户首次接触 记录、观察、引导
采纳阶段 Adoption 用户开始使用 支持答疑、收集反馈
保留阶段 Retention 用户持续使用 深化价值、提升体验
推荐阶段 Advocacy 用户推荐他人 收集成功案例、请求推荐
